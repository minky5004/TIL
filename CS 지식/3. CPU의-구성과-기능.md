# CPU의 구성과 기능

CPU(Central Processing Unit)는 프로그램 명령어를 해석하고 실행하는 컴퓨터의 두뇌다.
전원이 켜지는 순간부터 꺼질 때까지 **명령어 사이클**을 반복하며 프로그램을 실행한다.

---

## CPU의 3가지 구성 요소

```
┌──────────────────────────────────────────────┐
│                    CPU                       │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌───────────┐  │
│  │   ALU    │  │    CU    │  │ Registers │  │
│  └──────────┘  └──────────┘  └───────────┘  │
│        ↕              ↕              ↕       │
└──────────────────────────────────────────────┘
   연산 장치        제어 장치        레지스터
                        ↕
              [ 시스템 버스 — 메모리 / 입출력 장치 ]
```

### 1. 연산 장치 (ALU, Arithmetic Logic Unit)

산술 연산(덧셈·뺄셈·곱셈·나눗셈)과 논리 연산(AND·OR·NOT·XOR·비교)을 담당한다.
레지스터에서 피연산자를 받아 연산 후 결과를 다시 레지스터에 저장한다.

```java
// ALU가 수행하는 연산을 Java로 표현한 예시
public class SimpleALU {
    public static int execute(String opcode, int a, int b) {
        return switch (opcode) {
            case "ADD"  -> a + b;         // 산술 연산
            case "SUB"  -> a - b;
            case "AND"  -> a & b;         // 논리 연산
            case "OR"   -> a | b;
            case "XOR"  -> a ^ b;
            case "CMP"  -> Integer.compare(a, b); // 비교: -1, 0, 1
            default     -> throw new IllegalArgumentException("알 수 없는 연산: " + opcode);
        };
    }

    public static void main(String[] args) {
        System.out.println(SimpleALU.execute("ADD", 5, 3));  // 8
        System.out.println(SimpleALU.execute("AND", 0b1010, 0b1100));  // 8 (0b1000)
        System.out.println(SimpleALU.execute("CMP", 5, 3));  // 1 (5 > 3)
    }
}
```

### 2. 제어 장치 (Control Unit, CU)

명령어를 해독하고, 각 장치(ALU·레지스터·메모리·입출력 장치)에 제어 신호를 보낸다.
실제로 연산을 수행하지 않고, **"무엇을 언제 어떻게 할지"를 지시하는 지휘자** 역할이다.

| 역할 | 설명 |
|---|---|
| 명령어 해독 | IR에 저장된 명령어의 OpCode를 분석해 수행할 작업 결정 |
| 제어 신호 발생 | ALU·레지스터·메모리 등에 동작 신호 전달 |
| 데이터 흐름 관리 | CPU 내부와 외부 장치 사이의 데이터 이동 순서 조율 |

### 3. 레지스터 (Register)

CPU 내부의 고속 저장 공간이다. 용량은 매우 작지만 접근 속도가 가장 빠르다.
현재 처리 중인 명령어·주소·데이터를 임시로 보관한다.

| 레지스터 | 약어 | 역할 |
|---|---|---|
| 기억장치 주소 레지스터 | MAR | 접근할 메모리의 주소를 저장. "어디를 읽을지" |
| 기억장치 버퍼 레지스터 | MBR (= MDR) | 메모리에서 읽어온 데이터나 메모리에 쓸 데이터를 임시 저장. CPU-메모리 사이의 양방향 버퍼 |
| 명령어 레지스터 | IR | 현재 실행 중인 명령어를 저장. 제어 장치가 이 레지스터를 읽어 해독함 |
| 프로그램 카운터 | PC | 다음에 인출할 명령어의 메모리 주소를 저장. CPU의 내비게이션 역할 |

> **MBR = MDR**: Memory Buffer Register와 Memory Data Register는 같은 레지스터를 부르는 두 가지 공식 명칭이다. 교재마다 표기가 다르므로 혼동하지 말 것.

#### 범용 레지스터 (General Purpose Register, GPR)

위의 4개는 **특수 목적 레지스터**다. 이와 별도로 프로그래머가 직접 사용하는 **범용 레지스터**가 있다.

| 레지스터 | 주 용도 |
|---|---|
| 누산기 (Accumulator) | 산술·논리 연산의 기본 입출력 레지스터. ALU 연산 결과가 기본적으로 저장됨 |
| 스택 포인터 (Stack Pointer, SP) | 현재 스택의 최상단 주소를 가리킴. 함수 호출·반환 시 자동으로 변경됨 |
| 베이스 레지스터 (Base Register) | 간접 주소 지정에 사용. 배열·구조체의 시작 주소를 보관 |
| 인덱스 레지스터 (Index Register) | 배열 요소 접근 등 반복 연산에서 오프셋을 저장 |

> 현대 CPU(x86-64)는 `rax`, `rbx`, `rcx`, `rdx`, `rsi`, `rdi` 등 16개의 64비트 범용 레지스터를 제공한다. Java 코드를 작성할 때 직접 다루지는 않지만, JVM이 바이트코드를 네이티브 코드로 컴파일(JIT)할 때 이 레지스터들을 적극적으로 활용한다.

## CPU 클럭 (Clock)

CPU가 동작하는 박자(타이밍 신호)다. 클럭 한 번이 "틱(tick)"이며, 초당 틱 수를 **클럭 속도(Clock Speed)** 라 한다.

```
클럭 신호:  ▔▁▔▁▔▁▔▁▔▁▔▁  (주기적 펄스)
            ↑ 각 상승 엣지마다 CPU 내부 회로가 한 단계씩 동작
```

| 단위 | 의미 | 예시 |
|---|---|---|
| Hz | 초당 1사이클 | (역사적 초기 CPU) |
| MHz | 초당 100만 사이클 | 386 CPU ~33 MHz |
| GHz | 초당 10억 사이클 | 현대 CPU 3~5 GHz |

### 클럭과 명령어 사이클의 관계

명령어 하나를 처리하는 데 몇 번의 클럭이 필요한지를 **CPI(Cycles Per Instruction)** 라 하고,
한 클럭에 처리하는 명령어 수를 **IPC(Instructions Per Cycle)** 라 한다. 둘은 역수 관계다.

```
CPU 성능 = 클럭 속도(GHz) × IPC  =  클럭 속도(GHz) / CPI

예: 3 GHz CPU, CPI=1 (IPC=1.0) → 초당 30억 개 명령어 처리 (이상적)
    3 GHz CPU, CPI=3 (IPC=0.33) → 초당 10억 개 명령어 처리
```

> 같은 GHz라도 아키텍처가 다르면 성능이 크게 다를 수 있다. Apple M 시리즈가 Intel보다 낮은 GHz에서도 높은 성능을 내는 이유가 바로 **높은 IPC** 때문이다.

### Base Clock vs Boost Clock

| 구분 | 설명 |
|---|---|
| Base Clock | 기본 작동 클럭. 발열·전력 제한 내에서 항상 유지 가능한 속도 |
| Boost Clock (Turbo) | 온도와 전력에 여유가 있을 때 자동으로 상승하는 최대 클럭 |

## 명령어 사이클 (Instruction Cycle)

CPU가 하나의 명령어를 처리하는 과정이다. 전원이 켜지면 이 사이클을 무한 반복한다.

```
┌─────────────┐
│  1. Fetch   │  PC → MAR → 메모리 읽기 → MBR → IR    (명령어 인출)
└──────┬──────┘
       ↓
┌─────────────┐
│  2. Decode  │  IR의 OpCode를 제어장치(CU)가 해독     (명령어 해독)
└──────┬──────┘
       ↓
┌─────────────┐
│  3. Execute │  ALU 연산 / 메모리 접근 / 입출력 / 분기 (실행)
└──────┬──────┘
       ↓
┌─────────────┐
│  4. Check   │  인터럽트 신호 확인 → ISR 실행 후 복귀  (인터럽트 체크)
└──────┬──────┘
       ↓
    (반복)
```

### 단계별 세부 동작

**① 인출 단계 (Fetch)**

1. PC에 저장된 주소를 MAR로 복사
2. 제어 장치가 메모리 읽기 신호를 발생
3. 해당 주소의 명령어가 MBR로 전달됨
4. MBR의 내용이 IR로 복사
5. PC 값이 증가 (+1 또는 명령어 크기만큼) → 다음 명령어 주소 준비

**② 해독 단계 (Decode)**

1. IR의 명령어를 제어 장치의 **명령어 해독기**가 분석
2. OpCode를 보고 어떤 연산인지 판단 (산술 / 메모리 접근 / 분기 / 입출력)
3. 피연산자 주소가 있으면 MAR로 전달

**③ 실행 단계 (Execute)**

명령어 종류에 따라 동작이 달라진다.

| 명령어 종류 | 동작 |
|---|---|
| 산술/논리 연산 | ALU가 연산을 수행하고 결과를 레지스터에 저장 |
| 메모리 읽기 | MAR에 주소 설정 → 메모리 → MBR → 레지스터 |
| 메모리 쓰기 | 레지스터 값을 MBR에 복사 → MBR → 메모리 |
| 분기(Jump/Branch) | 조건이 참이면 PC에 분기 대상 주소를 덮어씀 |

**④ 인터럽트 처리 단계**

실행 단계가 끝난 뒤 CPU는 인터럽트 요청(IRQ) 신호를 확인한다. 인터럽트가 있으면 다음 순서로 처리한다.

```
현재 프로세스 실행 중
        ↓  인터럽트 발생 (키보드 입력, 타이머, I/O 완료 등)
① 컨텍스트 저장 (Context Save)
   → PC, 레지스터 값, 상태 플래그를 스택(또는 PCB)에 저장
        ↓
② 인터럽트 벡터 테이블 (IVT) 참조
   → 인터럽트 번호로 ISR(Interrupt Service Routine) 주소를 찾음
        ↓
③ ISR 실행
   → 해당 인터럽트를 처리하는 루틴 수행
        ↓
④ 컨텍스트 복원 (Context Restore)
   → 저장해 둔 레지스터·PC를 원래대로 복원
        ↓
⑤ IRET 명령어
   → 인터럽트 이전 실행 지점으로 정확히 복귀
```

> Java의 `Thread.interrupt()`는 이 하드웨어 인터럽트를 소프트웨어 수준에서 모방한 메커니즘이다. JVM은 해당 스레드의 **인터럽트 플래그**를 세우고, 스레드가 `sleep()`·`wait()` 등 대기 상태에 있을 때 `InterruptedException`을 던져 실행 흐름을 끊는다. 하드웨어 인터럽트처럼 즉시 강제 중단되는 것이 아니라 스레드가 스스로 플래그를 확인해야 동작한다는 점이 다르다.

```java
// 명령어 사이클을 Java로 시뮬레이션한 예시
public class CPUSimulator {

    // 레지스터
    private int pc  = 0;   // 프로그램 카운터
    private int mar = 0;   // 기억장치 주소 레지스터
    private int mbr = 0;   // 기억장치 버퍼 레지스터
    private int ir  = 0;   // 명령어 레지스터
    private int acc = 0;   // 누산기 (일반 레지스터)

    // 가상 메모리 (주소 → 명령어/데이터)
    private final int[] memory = {
        0x1005,  // [0] LOAD  addr=5  : 메모리 5번지의 값을 누산기로 로드
        0x2006,  // [1] ADD   addr=6  : 메모리 6번지 값을 누산기에 더함
        0x3007,  // [2] STORE addr=7  : 누산기 값을 메모리 7번지에 저장
        0x0000,  // [3] HALT           : 정지
        0,       // [4] (사용 안 함)
        10,      // [5] 데이터: 10
        20,      // [6] 데이터: 20
        0        // [7] 결과 저장 공간
    };

    public void run() {
        while (true) {
            // ── 1. Fetch ──
            mar = pc;
            mbr = memory[mar];
            ir  = mbr;
            pc++;

            int opcode  = (ir >> 12) & 0xF; // 상위 4비트: 연산 코드
            int operand = ir & 0x0FFF;       // 하위 12비트: 피연산자 주소

            // ── 2. Decode & Execute ──
            switch (opcode) {
                case 0x1 -> {  // LOAD: 메모리 → 누산기
                    mar = operand;
                    mbr = memory[mar];
                    acc = mbr;
                    System.out.printf("LOAD  mem[%d]=%d → acc=%d%n", operand, mbr, acc);
                }
                case 0x2 -> {  // ADD: 누산기 + 메모리
                    mar = operand;
                    mbr = memory[mar];
                    acc += mbr;  // ALU 연산
                    System.out.printf("ADD   mem[%d]=%d → acc=%d%n", operand, mbr, acc);
                }
                case 0x3 -> {  // STORE: 누산기 → 메모리
                    mar = operand;
                    mbr = acc;
                    memory[mar] = mbr;
                    System.out.printf("STORE acc=%d → mem[%d]%n", acc, operand);
                }
                case 0x0 -> {  // HALT
                    System.out.println("HALT — 실행 종료");
                    return;
                }
            }
        }
    }

    public static void main(String[] args) {
        new CPUSimulator().run();
        // LOAD  mem[5]=10 → acc=10
        // ADD   mem[6]=20 → acc=30
        // STORE acc=30 → mem[7]
        // HALT — 실행 종료
    }
}
```

## 파이프라이닝 (Pipelining) — 현대 CPU의 실행 방식

위에서 설명한 순차적 사이클은 **학습용 모델**이다. 현대 CPU는 **파이프라이닝** 기법으로 여러 명령어를 동시에 처리해 성능을 높인다.

```
순차 실행 (고전 방식):
명령어1: [Fetch][Decode][Execute]
명령어2:                          [Fetch][Decode][Execute]
명령어3:                                                   [Fetch][Decode][Execute]

파이프라인 실행 (현대 방식):
명령어1: [Fetch][Decode][Execute]
명령어2:        [Fetch][Decode][Execute]
명령어3:               [Fetch][Decode][Execute]
                 ↑ 동시에 진행 → 처리량 향상
```

### 파이프라인 해저드 (Hazard)

파이프라인을 방해하는 3가지 상황이 있다.

| 해저드 종류 | 원인 | 예시 |
|---|---|---|
| 데이터 해저드 | 앞 명령어의 결과를 뒤 명령어가 즉시 필요로 함 | `r1 = r2 + r3` 직후 `r4 = r1 + 1` |
| 제어 해저드 | 분기 명령어로 인해 다음 명령어를 예측하기 어려움 | `if/while` 분기문 |
| 구조적 해저드 | 두 명령어가 같은 하드웨어 자원을 동시에 필요로 함 | 메모리 읽기·쓰기 충돌 |

> 해저드가 발생하면 CPU는 **파이프라인 버블(stall)** 을 삽입해 실행을 잠시 지연시킨다. 현대 CPU는 **투기적 실행(Speculative Execution)** 으로 분기 결과를 예측해 먼저 실행하는 방식으로 제어 해저드를 완화한다.

### 분기 예측 (Branch Prediction)

제어 해저드를 완화하기 위해 CPU가 분기 결과를 미리 예측하고 투기적으로 명령어를 실행하는 기법이다.

- 현대 CPU의 분기 예측 정확도: **99% 이상** (SPEC 2017 벤치마크 기준)
- 예측 실패(Misprediction) 시 파이프라인을 비우고 다시 시작 → **최대 수십 사이클 낭비**

```java
import java.util.Arrays;
import java.util.Random;

// 분기 예측이 Java 성능에 미치는 영향 예시
public class BranchPredictionDemo {
    public static void main(String[] args) {
        int SIZE = 1_000_000;
        int[] data = new int[SIZE];
        Random rand = new Random();
        for (int i = 0; i < SIZE; i++) data[i] = rand.nextInt(256);

        // ── 정렬되지 않은 배열: 분기 예측 실패 多 ──
        long start = System.nanoTime();
        long sum = 0;
        for (int v : data) {
            if (v >= 128) sum += v;  // CPU가 다음 분기를 예측하기 어려움
        }
        long unsortedTime = System.nanoTime() - start;

        // ── 정렬된 배열: 분기 패턴이 명확 → 예측 성공률 ↑ ──
        Arrays.sort(data);
        start = System.nanoTime();
        sum = 0;
        for (int v : data) {
            if (v >= 128) sum += v;  // 128 미만은 항상 false, 이상은 항상 true → 예측 쉬움
        }
        long sortedTime = System.nanoTime() - start;

        System.out.printf("정렬 전: %,d ns%n", unsortedTime);
        System.out.printf("정렬 후: %,d ns%n", sortedTime);
        // 정렬 후가 수 배 빠름 — 같은 코드인데 분기 예측 성공률 차이만으로 발생
    }
}
```

> 이 예제는 2012년 Stack Overflow에서 화제가 된 **"정렬된 배열이 정렬되지 않은 배열보다 빠른 이유"** 질문의 핵심이다. 코드 로직은 동일하지만 분기 예측 성공률 차이 하나로 수 배의 성능 차이가 발생한다.

---

## 실제 사고 사례

**Spectre·Meltdown (2018년 공개)**

투기적 실행(Speculative Execution)의 부작용으로 발생한 보안 취약점이다. 분기 예측이 틀렸을 때 CPU가 이미 투기적으로 실행한 명령어의 결과는 레지스터에 반영되지 않지만, **캐시 상태는 변경된 채로 남는다.** 공격자는 캐시 접근 시간 차이를 측정해 커널 메모리나 다른 프로세스의 비밀 데이터를 읽을 수 있었다.

- 영향 범위: Intel, AMD, ARM 기반 CPU 대부분 (사실상 전 세계 PC·서버·스마트폰)
- 수정 방법: OS 패치(커널 페이지 테이블 분리) + 마이크로코드 업데이트 → 일부 CPU에서 최대 30% 성능 저하
- 교훈: 파이프라인의 성능 최적화 기법이 의도치 않은 **사이드 채널(side channel)** 을 만들 수 있다

## 핵심 정리

| 개념 | 한 줄 요약 |
|---|---|
| ALU | 산술·논리 연산을 수행하는 계산기 |
| 제어 장치 (CU) | 명령어를 해독하고 각 장치에 신호를 내리는 지휘자 |
| MAR | "어디를" — 접근할 메모리 주소 저장 |
| MBR (= MDR) | "무엇을" — 메모리와 주고받는 데이터 임시 보관 |
| IR | "지금 무엇을 하고 있는가" — 현재 명령어 저장 |
| PC | "다음에 무엇을 할 것인가" — 다음 명령어 주소 저장 |
| 명령어 사이클 | Fetch → Decode → Execute → (인터럽트 체크) 반복 |
| 인터럽트 처리 | 컨텍스트 저장 → IVT 참조 → ISR 실행 → 컨텍스트 복원 → IRET |
| 클럭 속도 | 초당 사이클 수(GHz). 클럭 × IPC = 실제 성능 |
| CPI / IPC | CPI = 명령어당 클럭 수(작을수록 빠름), IPC = 클럭당 명령어 수(클수록 빠름). 역수 관계 |
| 파이프라이닝 | 여러 명령어를 겹쳐서 실행해 처리량을 높이는 기법 |
| 파이프라인 해저드 | 파이프라인 실행을 방해하는 데이터·제어·구조적 충돌 |
| 분기 예측 | CPU가 분기 결과를 미리 예측해 투기적으로 실행 — 오예측 시 파이프라인 비움 |
