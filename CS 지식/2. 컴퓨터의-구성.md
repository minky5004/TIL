# 컴퓨터의 구성

컴퓨터는 **하드웨어(물리 장치)** 와 **소프트웨어(프로그램)** 두 축으로 이루어진다.
이 문서는 각 구성 요소가 무엇인지보다, **왜 이렇게 나뉘었는가**와 **구성 요소끼리 어떻게 협력하는가**에 집중한다.

---

## 하드웨어

### CPU (중앙 처리 장치)

컴퓨터의 두뇌. **연산(ALU)** 과 **제어(CU)** 를 담당한다.

| 내부 구성 | 역할 |
|---|---|
| ALU (산술논리 연산 장치) | 덧셈·뺄셈·비교·논리 연산 수행 |
| CU (제어 장치) | 명령어를 해석하고 각 장치에 신호 전달 |
| 레지스터 | CPU 내부 임시 저장 공간. 가장 빠른 메모리 |

CPU는 전원이 켜진 순간부터 꺼질 때까지 아래 **명령어 사이클**을 무한 반복한다.

```
① Fetch      PC(Program Counter)가 가리키는 주소에서 명령어를 메모리로부터 가져와
             IR(Instruction Register)에 저장. 이후 PC += 1 (다음 명령어 준비)

② Decode     IR의 명령어를 해독 — 어떤 연산인지(opcode), 피연산자가 무엇인지 파악

③ Execute    ALU가 연산 수행. 메모리 접근이 필요하면 시스템 버스를 통해 Read/Write 발생

④ Write-back 연산 결과를 레지스터 또는 메모리에 기록 (필요한 경우에만)
```

> **PC(Program Counter)** 는 "다음에 실행할 명령어의 주소"를 담는 레지스터다. `if`, `for`, 메서드 호출이 일어나면 PC가 해당 주소로 점프한다. Java에서 예외 스택 트레이스에 줄 번호가 나오는 것도 PC가 추적하는 실행 위치를 기반으로 한다.

### 기억 장치 계층 구조

빠를수록 용량이 작고 비싸다. 이 트레이드오프를 계층으로 해결한다.

```
                    ▲ 빠름 · 작음

            ┌──────────────┐
            │   레지스터    │  수 바이트    수 피코초
          ┌─┴──────────────┴─┐
          │     L1 캐시       │  16~128 KB   수 나노초
        ┌─┴────────────────────┴─┐
        │       L2 캐시            │  ~수 MB     수십 나노초
      ┌─┴──────────────────────────┴─┐
      │          L3 캐시               │  2~64 MB+  수십~수백 나노초
    ┌─┴────────────────────────────────┴─┐
    │       RAM  (주기억장치)               │  GB 단위   ~100 ns
  ┌─┴──────────────────────────────────────┴─┐
  │               SSD / HDD                   │  TB 단위   수십 μs ~ ms
  └────────────────────────────────────────────┘

                    ▼ 느림 · 큼
```

#### 캐시 메모리 — 왜 필요한가

CPU의 연산 속도와 RAM의 응답 속도 차이가 너무 크다. L3 캐시 미스 한 번이 약 100~300 클럭 낭비다. 만약 캐시가 없고 모든 데이터를 RAM에서 직접 읽는다면, 현재 컴퓨터보다 약 100배 느려진다.

| 캐시 레벨 | 크기 | 접근 속도 | 특징 |
|---|---|---|---|
| L1 | 16 ~ 128 KB | ~수 나노초 | 코어마다 전용, 가장 빠름 |
| L2 | 256 KB ~ 수 MB | ~수십 나노초 | 코어마다 전용, L1 미스 시 조회 |
| L3 | 2 ~ 64 MB+ | ~수십~수백 나노초 | 코어 간 공유, 마지막 캐시 관문 |

> CPU는 데이터를 찾을 때 **L1 → L2 → L3 → RAM** 순서로 탐색한다. 찾으면 **캐시 히트(Cache Hit)**, 못 찾으면 **캐시 미스(Cache Miss)** — 이때 RAM까지 내려가 데이터를 올려온다.

#### RAM (주기억 장치)

실행 중인 프로그램과 데이터를 임시 저장한다. **전원이 꺼지면 내용이 사라진다.**

Java 프로그램이 실행되면 JVM이 OS에게 RAM을 요청해 힙(Heap), 스택(Stack), 메타스페이스 등으로 분할해서 사용한다.

#### 보조 기억 장치 (SSD / HDD)

프로그램 코드, 파일, 데이터를 영구적으로 저장한다. 실행 시 RAM으로 적재(load)된 후에야 CPU가 접근할 수 있다.

### 입출력 장치 (I/O)

| 분류 | 예시 |
|---|---|
| 입력 | 키보드, 마우스, 마이크, 카메라, 터치스크린 |
| 출력 | 모니터, 스피커, 프린터 |
| 입출력 겸용 | USB 드라이브, 네트워크 카드, 터치스크린 |

I/O 장치는 CPU보다 훨씬 느리다. 이 속도 차이를 처리하는 방법이 두 가지 있다.

| 방식 | 동작 | 문제점 |
|---|---|---|
| **폴링(Polling)** | CPU가 I/O 완료 여부를 주기적으로 직접 확인 | CPU가 아무 일도 못 하고 계속 확인만 함 (바쁜 대기) |
| **인터럽트(Interrupt)** | I/O 장치가 완료되면 CPU에 신호를 보냄 | CPU는 그 사이 다른 일을 처리할 수 있음 |

인터럽트 처리 흐름:

```
 1. CPU가 정상적으로 명령어를 실행 중
 2. I/O 장치(키보드, 디스크 등)가 "작업 완료" 인터럽트 신호 전송
 3. CPU는 현재 실행 상태(레지스터, PC)를 스택에 저장
 4. ISR(인터럽트 서비스 루틴) 실행 — 해당 I/O 처리
 5. 저장해 둔 상태를 복원 → 하던 작업 재개
```

> 키보드 키 하나를 누를 때마다 CPU가 "키가 눌렸나?" 를 매 클럭마다 확인한다면 CPU를 낭비하게 된다. 인터럽트 덕분에 CPU는 키 입력이 올 때만 잠깐 반응하고 나머지 시간은 다른 연산에 집중할 수 있다.

### DMA (Direct Memory Access)

인터럽트는 "완료 알림"을 해결했지만, **대용량 데이터 이동** 문제가 남는다.
예를 들어 디스크에서 1GB 파일을 RAM으로 복사할 때 CPU가 직접 바이트 하나씩 옮기면 CPU가 그 시간 동안 완전히 묶인다.

**DMA 컨트롤러**는 CPU를 대신해 메모리 ↔ I/O 장치 간 데이터를 직접 전송한다.

```
   [DMA 없을 때]                       [DMA 있을 때]
   디스크 → CPU → RAM                  디스크 → RAM (DMA 컨트롤러가 처리)
   (CPU가 중간에서 일일이 중계)          CPU는 그동안 다른 일 가능
                                        완료되면 인터럽트로 CPU에 통보
```

> 현대 컴퓨터에서 디스크 읽기, 네트워크 패킷 수신, 그래픽 렌더링은 모두 DMA로 처리된다. CPU는 "시작"과 "완료 처리"에만 개입한다.

**Java NIO와 DMA의 연결 고리**

Java NIO의 `ByteBuffer.allocateDirect()`는 JVM 힙 바깥의 OS 네이티브 메모리를 직접 할당한다. 이 영역은 GC 대상이 아니며, OS가 DMA를 통해 디스크·네트워크 데이터를 이 버퍼에 직접 쓸 수 있어 복사 횟수가 줄어든다.

```java
import java.nio.ByteBuffer;
import java.nio.channels.FileChannel;
import java.nio.file.Path;
import java.nio.file.StandardOpenOption;

// 힙 버퍼: JVM 힙에 할당 → GC 관리, OS → JVM 복사 발생
ByteBuffer heapBuffer   = ByteBuffer.allocate(4096);

// 다이렉트 버퍼: OS 네이티브 메모리에 할당 → DMA가 직접 이 버퍼에 쓸 수 있음
ByteBuffer directBuffer = ByteBuffer.allocateDirect(4096);

// 파일을 다이렉트 버퍼로 읽기 — 복사 없이 OS → directBuffer 직접 전달
try (FileChannel channel = FileChannel.open(
        Path.of("large_file.bin"), StandardOpenOption.READ)) {
    int bytesRead = channel.read(directBuffer);  // DMA 경로 활용
    directBuffer.flip();
    // directBuffer 처리...
}
// 주의: directBuffer는 GC가 관리하지 않으므로 수동 해제가 필요하다
// 잘못 쓰면 오히려 성능이 나빠질 수 있으므로 대용량 I/O에만 사용한다
```

## 소프트웨어

하드웨어를 제어하는 프로그램 전체를 말한다.

| 종류 | 설명 | 예시 |
|---|---|---|
| **응용 소프트웨어** | 사용자가 목적을 위해 실행 | 메모장, 웹 브라우저, 게임, IntelliJ |
| **시스템 소프트웨어** | 하드웨어와 응용 SW가 원활히 동작하도록 지원 | OS(운영체제), 컴파일러, 디바이스 드라이버 |

> 응용 소프트웨어는 하드웨어에 직접 접근하지 않는다. 반드시 **OS(운영체제)를 통해** 요청한다. OS가 중간에서 자원을 관리하고 보호하기 때문이다.

## 시스템 버스

CPU, 메모리, I/O 장치를 물리적으로 연결하는 선(wire)의 묶음이다. 역할에 따라 세 가지로 나뉜다.

| 버스 종류 | 방향 | 역할 |
|---|---|---|
| **데이터 버스** | 양방향 | CPU ↔ 메모리/IO 간 실제 데이터 전송 |
| **주소 버스** | 단방향 (CPU → 메모리/IO) | CPU가 읽거나 쓸 위치(주소) 전달 |
| **제어 버스** | 양방향 | READ/WRITE 명령, READY/WAIT 상태 응답 |

```
     CPU
      │
      ├── 주소 버스 (단방향) ──────────────→ 메모리 / I/O
      │
      ├── 데이터 버스 (양방향) ←──────────→ 메모리 / I/O
      │
      └── 제어 버스 (양방향) ←──────────→ 메모리 / I/O
```

### 주소 버스 너비와 최대 메모리 용량

주소 버스의 비트 수가 곧 표현 가능한 주소의 개수다.

```
32비트 주소 버스 → 2³² = 4,294,967,296개의 주소 → 최대 4 GB RAM
64비트 주소 버스 → 2⁶⁴ ≈ 16 엑사바이트 (사실상 무제한)
```

> 32비트 Windows가 RAM을 4GB까지밖에 인식하지 못했던 이유가 바로 이것이다. 주소 버스 폭은 소프트웨어로 우회할 수 없는 **하드웨어 한계**다. 2000년대 중반, 데이터베이스 서버와 영상 편집 워크스테이션은 이 한계에 매일 부딪혔고, 결국 64비트 전환이 이루어졌다.

## 데이터 읽기 과정

CPU가 메모리에서 데이터를 읽는 한 사이클을 버스 관점에서 본 흐름이다.

```
 ① 주소 전달 (주소 버스)
    CPU ──── "0x7F3A 주소의 데이터 읽어줘" ────→ 메모리

 ② 제어 신호 전달 (제어 버스)
    CPU ──── READ 신호 ────────────────────→ 메모리

 ③ 데이터 응답 (데이터 버스)
    CPU ←─── 해당 주소의 데이터 ────────────── 메모리

 ④ 완료 신호 (제어 버스)
    CPU ←─── READY 신호 ─────────────────── 메모리
```

## Java 코드 — 하드웨어와의 연결 고리

### JVM이 OS로부터 RAM을 가져오는 과정

```java
// JVM은 시작 시 OS에 RAM을 요청한다.
// Runtime API로 현재 JVM이 확보한 메모리 상태를 조회할 수 있다.
long maxMemory   = Runtime.getRuntime().maxMemory();    // JVM이 사용할 수 있는 최대 RAM
long totalMemory = Runtime.getRuntime().totalMemory();  // 현재 JVM에 할당된 RAM
long freeMemory  = Runtime.getRuntime().freeMemory();   // 현재 힙의 여유 공간

System.out.printf("최대  : %,d bytes (%.1f MB)%n", maxMemory,   maxMemory   / 1e6);
System.out.printf("할당됨: %,d bytes (%.1f MB)%n", totalMemory, totalMemory / 1e6);
System.out.printf("여유  : %,d bytes (%.1f MB)%n", freeMemory,  freeMemory  / 1e6);
// 결과 예시)
// 최대  : 4,294,967,296 bytes (4294.9 MB)  ← -Xmx 설정 또는 OS 여유 메모리 기반
// 할당됨:   268,435,456 bytes ( 268.4 MB)  ← 현재 OS에서 가져온 힙 크기
// 여유  :   262,144,000 bytes ( 262.1 MB)  ← 아직 객체에 사용되지 않은 공간
```

### 캐시 히트 vs 미스 — 접근 순서가 성능을 좌우한다

```java
int SIZE = 1024 * 1024;  // 1M 정수 배열
int[] arr = new int[SIZE];

// 캐시 친화적 접근: 순차 읽기 (행 우선)
// 인접 메모리를 연속으로 읽으면 캐시 라인 전체가 한번에 올라와 캐시 히트 연속 발생
long start = System.nanoTime();
long sum = 0;
for (int i = 0; i < SIZE; i++) {
    sum += arr[i];  // 순서대로 읽음 → L1/L2 캐시 히트
}
long sequential = System.nanoTime() - start;

// 캐시 비친화적 접근: 큰 보폭으로 건너뛰며 읽기
// 매 접근마다 새 캐시 라인을 RAM에서 올려야 하므로 캐시 미스 반복
start = System.nanoTime();
sum = 0;
for (int i = 0; i < SIZE; i += 64) {  // 256바이트씩 건너뜀
    sum += arr[i];  // 매번 캐시 미스 → RAM까지 내려감
}
long strided = System.nanoTime() - start;

System.out.printf("순차 접근: %,d ns%n", sequential);
System.out.printf("건너뜀 접근: %,d ns%n", strided);
// 순차 접근이 수 배에서 수십 배 빠르게 나온다
```

> 루프의 순서 하나만 바꿔도 캐시 효율이 극적으로 달라진다. 2차원 배열을 행 우선으로 접근하면 순차 메모리 접근이 되어 캐시 히트가 연속 발생한다. 열 우선으로 접근하면 매번 캐시 미스가 발생하여 성능이 급격히 떨어진다.

## 실제 사고 사례

- **32비트 4GB 한계**: 2000년대 중반, 32비트 Windows 서버를 운영하던 데이터베이스 서버들이 RAM 4GB 한계에 부딪혀 더 이상 메모리를 추가해도 인식하지 못하는 문제가 광범위하게 발생. 주소 버스 너비라는 하드웨어 제약이 소프트웨어로 해결 불가능하다는 것을 보여준 사례. 이후 64비트 전환을 촉진시켰다.

- **캐시 미스 연쇄 장애**: 2021년 Fastly CDN은 고객의 잘못된 캐시 설정 하나가 연쇄 반응을 일으켜 Reddit, GitHub, 뉴욕타임즈 등 주요 사이트들이 약 1시간 동안 전세계적으로 접속 불가 상태가 된 사건. 캐시 계층이 무너졌을 때 하위 서버(오리진)에 모든 요청이 폭주하는 **캐시 스탬피드(Cache Stampede)** 패턴의 대표 사례.

- **Intel Meltdown 취약점 (2018)**: L3 캐시의 타이밍 차이를 이용해 다른 프로세스의 커널 메모리를 훔쳐볼 수 있는 취약점. 캐시 히트 시 응답이 빠르다는 하드웨어 특성 자체가 보안 구멍이 된 사례. 패치 이후 일부 서버의 I/O 성능이 최대 30% 하락했다.
