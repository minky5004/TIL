# 데이터와 표현

## 비트와 바이트

컴퓨터는 전기 신호로 동작한다. 전압이 기준보다 높으면 1, 낮으면 0 — 이것이 2진법의 시작이다.

- **비트(bit)**: 0 또는 1을 담는 최소 단위
- **바이트(byte)**: 8비트 묶음. 2⁸ = 256가지 표현 가능
- 단위 체계: `Byte → KB → MB → GB → TB`

> 프로그래밍 관행상 1KB = 1,024바이트로 사용하지만, SI 공식 표준에서는 1KB = 1,000바이트다. 1,024배 단위의 정확한 명칭은 **KiB(키비바이트)·MiB·GiB**다. HDD 제조사는 SI 표준(1,000배)으로 표기하고 OS는 관행(1,024배)으로 표시하기 때문에, 1TB HDD를 꽂으면 Windows에서 931GB로 보이는 차이가 발생한다.

## 정수 표현

### 부호 비트

최상위 비트(MSB)가 부호를 결정한다.

- `0xxx...` → 양수 또는 0
- `1xxx...` → 음수

### 2의 보수 (Two's Complement)

음수를 저장하는 방식. **"모든 비트를 반전한 뒤 1을 더한다"** 는 규칙이다.

```
예) 3 (= 0011) 의 음수 구하기
1단계 반전: 0011 → 1100
2단계 +1 :  1100 + 1 = 1101  →  -3
```

**왜 2의 보수를 쓰는가?** 덧셈 회로 하나로 뺄셈까지 처리할 수 있기 때문이다.
3 + (−3)을 계산하면 자동으로 0이 나온다.

| 비트 수 | 표현 범위 |
|---|---|
| 8비트 | -128 ~ 127 |
| 16비트 | -32,768 ~ 32,767 |
| 32비트 | -2,147,483,648 ~ 2,147,483,647 |

n비트 범위: −2ⁿ⁻¹ ~ 2ⁿ⁻¹ − 1

### 정수 오버플로우 (Integer Overflow)

n비트 정수의 최댓값에서 1을 더하면 최솟값으로 **넘어간다(wrap-around)**.

```
8비트 예시:
127 + 1 = -128   (01111111 + 00000001 = 10000000)
```

**실제 사고 사례:**
- **팩맨 256레벨 버그**: 레벨을 8비트 정수로 저장했다. 256 도달 시 0으로 오버플로우 되면서 화면이 깨지는 현상 발생
- **쇼핑카트 음수 결제**: 장바구니 합계가 오버플로우 → 음수가 되어 결제 금액이 0이 되는 취약점
- **유튜브 조회수**: 강남스타일 조회수가 `int` 최댓값(약 21억)에 근접하자, 유튜브가 2014년 조회수 카운터를 `long`으로 전환

**Java 코드:**

```java
int max = Integer.MAX_VALUE;    // 2,147,483,647
System.out.println(max + 1);   // -2,147,483,648  ← wrap-around, 예외 없이 조용히 발생

// Java 8+: 오버플로우 시 ArithmeticException을 던지는 안전한 버전
Math.addExact(Integer.MAX_VALUE, 1);  // throws ArithmeticException: integer overflow

// 범위가 부족하면 long으로 올려야 한다
long safe = (long) max + 1;     // 2,147,483,648  정상
```

> Java는 정수 오버플로우가 발생해도 **기본적으로 예외를 던지지 않는다**. 조용히 wrap-around 되기 때문에 버그를 발견하기 어렵다. 정밀도가 중요한 연산엔 `Math.addExact()` 계열을 사용하라.

## 실수 표현 (IEEE 754 부동소수점)

32비트 float의 구성:

```
[부호: 1비트] [지수 E: 8비트] [가수 M: 23비트]
```

값은 `±1.M × 2^(E - bias)` 형태로 저장된다. 지수 비트에 저장된 값은 **실제 지수가 아니라** bias를 더한 값이다. 32비트 float의 bias = 127이므로, 지수 비트가 `10000000`(= 128)이면 실제 지수는 128 − 127 = **1**이다.

> **64비트 double** 구성: 부호(1) + 지수 E(11) + 가수 M(52), bias = 1023. 표현 범위가 훨씬 넓고 정밀도도 높다.

### IEEE 754 특수값

부동소수점에는 일반 숫자 외에 특수한 상태를 나타내는 값이 정의되어 있다.

| 값 | 지수 비트 | 가수 비트 | 발생 상황 |
|---|---|---|---|
| +∞ / −∞ | 모두 1 | 모두 0 | `1.0 / 0.0`, 오버플로우 |
| NaN (Not a Number) | 모두 1 | 0이 아님 | `0/0`, `√-1`, `∞−∞` |
| 비정규화 수 (Denormal) | 모두 0 | 0이 아님 | 0에 매우 가까운 수 표현 |

NaN은 **자기 자신과도 같지 않다**는 특이한 성질이 있다.

**Java 코드:**

```java
double inf  =  1.0 / 0.0;    // Infinity
double ninf = -1.0 / 0.0;    // -Infinity
double nan  =  0.0 / 0.0;    // NaN

System.out.println(nan == nan);               // false  ← NaN은 자신과도 다르다
System.out.println(Double.isNaN(nan));        // true   ← 반드시 이 방법으로 검사
System.out.println(Double.isInfinite(inf));   // true

// NaN은 전파된다: NaN이 연산에 끼어들면 결과도 NaN
System.out.println(nan + 1);    // NaN
System.out.println(nan > 0);    // false
System.out.println(nan < 0);    // false  ← 모든 비교가 false
```

### 오차 문제

실수는 2진수로 **정확히 표현할 수 없는 경우**가 많다. 저장 시 근사값으로 반올림되기 때문이다.

```python
>>> 0.1 + 0.2
0.30000000000000004   # Python
```

```java
System.out.println(0.1 + 0.2);   // 0.30000000000000004  Java도 동일한 오차
System.out.println(1.0 - 0.9);   // 0.09999999999999998
```

**언제 문제가 되는가?** 금융 계산, 과학 시뮬레이션처럼 정밀도가 중요한 경우 오차가 누적되어 결과가 틀릴 수 있다.

**대안:**

| 방식 | 설명 | 사용 예 |
|---|---|---|
| Decimal | 10진 임의 정밀도 계산 | Python `decimal` 모듈, Java `BigDecimal` |
| Fixed-point | 소수점 위치를 고정해 정수처럼 처리 | 임베디드, 금융 시스템 |

**Java 코드 — BigDecimal로 금융 계산:**

```java
// 잘못된 방법: double 생성자 사용 → 이미 오차가 존재한 채로 BigDecimal이 만들어진다
BigDecimal wrong = new BigDecimal(0.1);
// 결과: 0.1000000000000000055511151231257827021181583404541015625

// 올바른 방법: 반드시 String으로 생성
BigDecimal a = new BigDecimal("0.1");
BigDecimal b = new BigDecimal("0.2");
System.out.println(a.add(b));              // 0.3  정확

// 금액 계산 실무 예시
BigDecimal price    = new BigDecimal("19900");
BigDecimal quantity = new BigDecimal("3");
BigDecimal total    = price.multiply(quantity);  // 59700
```

> 금융 도메인에서 `double`로 금액을 계산하는 것은 버그다. 항상 `BigDecimal`을 사용하고, 생성할 때는 **String 생성자**를 써야 한다.

## 문자 표현

### 아스키 코드 (ASCII)

7비트로 128개 문자를 정의한 초기 표준. 컴퓨터 내부에서는 상위 1비트를 0으로 채워 **1바이트(8비트)** 단위로 저장한다.

```
'A' = 65,  'a' = 97,  '0' = 48
```

**한계:** 영어와 특수문자만 담을 수 있어 다국어 표현이 불가능하다.

### 유니코드 (Unicode)

아스키의 한계를 극복하기 위해 **1991년** 처음 발표된 국제 문자 표준. 21비트로 약 111만 개 문자를 표현한다.

- 범위: `U+0000` ~ `U+10FFFF`
- 전 세계 모든 언어 + 이모지 포함

### UTF-8

유니코드를 **실제로 파일에 저장·전송하는 인코딩 방식**. 1~4바이트 가변 길이를 사용한다.

| 문자 종류 | 바이트 수 |
|---|---|
| 영어·숫자 (ASCII 범위) | 1바이트 |
| 유럽어·아랍어 등 | 2바이트 |
| 한국어·일본어·중국어 | 3바이트 |
| 희귀 문자·이모지 | 4바이트 |

**왜 UTF-8이 사실상 표준이 되었는가?** 자주 쓰는 영어·숫자를 1바이트로 처리해 용량이 작고, 기존 ASCII 파일과 완전히 호환되기 때문이다.

> **유니코드 vs UTF-8 구분:** 유니코드는 "각 문자에 번호를 부여하는 규칙"이고, UTF-8은 "그 번호를 실제 바이트로 변환하는 방법"이다. 같은 개념이 아니다.

**Java 코드 — 인코딩/디코딩:**

```java
import java.nio.charset.StandardCharsets;

String text = "안녕하세요";

// 인코딩: String → byte[]
byte[] utf8  = text.getBytes(StandardCharsets.UTF_8);
byte[] utf16 = text.getBytes(StandardCharsets.UTF_16);

System.out.println(utf8.length);   // 15  (한글 5자 × 3바이트)
System.out.println(utf16.length);  // 12  (한글 5자 × 2바이트 + BOM 2바이트)

// 디코딩: byte[] → String
String decoded = new String(utf8, StandardCharsets.UTF_8);
System.out.println(decoded);       // 안녕하세요

// charset 미지정 → Java 버전에 따라 결과가 다르다
byte[] risky = text.getBytes();
// Java 17 이하: OS 기본 인코딩 사용 (Windows = MS949, Linux = UTF-8)
// Java 18+  : UTF-8 고정 (JEP 400)
```

> Java 18(JEP 400)부터 `getBytes()` 기본 인코딩이 UTF-8로 고정되었다. 그러나 Java 17 이하 환경이나 `-Dfile.encoding` 옵션으로 오버라이드된 경우 동작이 달라지므로, **버전에 무관하게 `StandardCharsets.UTF_8`을 명시**하는 것이 안전하다.

### UTF-8 vs UTF-16 vs UTF-32

유니코드를 저장하는 인코딩 방식은 여럿이다.

| 인코딩 | 단위 크기 | 한글 한 글자 | 주 사용처 |
|---|---|---|---|
| UTF-8 | 가변 (1~4바이트) | 3바이트 | 웹, 파일 저장, Linux |
| UTF-16 | 가변 (2~4바이트) | 2바이트 | Java, JavaScript, Windows 내부 |
| UTF-32 | 고정 (4바이트) | 4바이트 | 처리 단순화가 필요한 내부 연산 |

한글만 저장하는 파일은 UTF-16이 UTF-8보다 크기가 작다. 그러나 웹에서는 영어·HTML 태그가 많아 UTF-8이 훨씬 효율적이다.

### BOM (Byte Order Mark)

파일 맨 앞에 붙는 특수 마커(`U+FEFF`). 어떤 인코딩과 바이트 순서로 저장됐는지 알려준다.

- **UTF-8 BOM**: `EF BB BF` → 유니코드 표준에서 **권장하지 않음**. 일부 파서가 오작동할 수 있다.
- **UTF-16 BOM**: 필수. `FE FF`(Big-endian) 또는 `FF FE`(Little-endian)

## 엔디안 (Endianness)

여러 바이트로 이루어진 데이터를 메모리에 저장할 때 **바이트를 어떤 순서로 놓는가**의 문제다.

숫자 `0x12345678`을 4바이트로 저장하면:

```
주소:          0x00  0x01  0x02  0x03
Big-endian:    12    34    56    78    ← 사람이 읽는 순서와 동일
Little-endian: 78    56    34    12    ← 낮은 자리 바이트가 앞에
```

| 방식 | 특징 | 사용처 |
|---|---|---|
| Big-endian | 높은 자리 바이트 → 낮은 주소 | 네트워크 프로토콜 (TCP/IP 표준) |
| Little-endian | 낮은 자리 바이트 → 낮은 주소 | x86, ARM (대부분의 PC·모바일) |

**왜 중요한가?** 네트워크는 Big-endian(= "네트워크 바이트 오더")을 표준으로 쓴다. x86 PC는 Little-endian이므로, 소켓 통신 시 반드시 바이트 순서를 변환해야 한다.

```c
uint16_t port = 8080;
htons(port)   // host to network short  →  Little-endian을 Big-endian으로
ntohs(port)   // network to host short  →  Big-endian을 Little-endian으로
```

**Java 코드 — ByteBuffer로 엔디안 제어:**

```java
import java.nio.ByteBuffer;
import java.nio.ByteOrder;
import java.nio.file.Files;
import java.nio.file.Path;

int value = 0x12345678;

// Big-endian으로 저장 (ByteBuffer 기본값, 네트워크 바이트 오더와 동일)
ByteBuffer bigBuf = ByteBuffer.allocate(4);
bigBuf.order(ByteOrder.BIG_ENDIAN).putInt(value);
// 메모리: 12 34 56 78

// Little-endian으로 저장 (x86 바이너리 파일 읽을 때)
ByteBuffer littleBuf = ByteBuffer.allocate(4);
littleBuf.order(ByteOrder.LITTLE_ENDIAN).putInt(value);
// 메모리: 78 56 34 12

// 실무 예시: 바이너리 파일에서 Little-endian int 읽기
byte[] rawBytes = Files.readAllBytes(Path.of("data.bin"));
int parsed = ByteBuffer.wrap(rawBytes)
                       .order(ByteOrder.LITTLE_ENDIAN)
                       .getInt();
```

> Java의 `ByteBuffer` 기본값은 **Big-endian**이다. C/C++로 작성된 x86 바이너리 파일을 읽을 때는 반드시 `ByteOrder.LITTLE_ENDIAN`으로 바꿔야 숫자가 올바르게 파싱된다.
